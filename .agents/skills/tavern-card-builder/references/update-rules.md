# [mvu_update] 变量更新规则条目写法

给"变量 AI"（或主 AI 的变量输出段）看的操作手册。写得越准，AI patch 越稳；写得越冗，token 越炸、AI 越复读。

## 1. 输出容器与方言声明

条目开头先锁死输出格式（RFC 标准方言示例）：

```
输出格式:
  容器结构:
    open_tag:  <UpdateVariable>
    inner_tag: <JSONPatch>
    payload:   一个 JSONPatch 数组
    close_tag: </JSONPatch></UpdateVariable>
  JSONPatch 操作:
    add:     { op: "add",     path: "/...", value: ... }
    replace: { op: "replace", path: "/...", value: ... }
    remove:  { op: "remove",  path: "/..." }
  数组尾部追加: path 以 /- 结尾
```

输出契约补充：正例照抄 + 四条绝对禁止（markdown fence 包 JSON / 整对象覆盖 / JSON 注释 / 裸大对象）；**无变化输出空数组 `[]` 而非省略块**；3+ 字段同变可整对象单次 replace。官方方言卡另有 `delta`（数值增减）/`insert`/`move`，见 SKILL.md 方言表——照抄该卡「变量输出格式」固定条目，别自创。

## 2. 章节骨架（三个成熟版本，按规模选）

**精简版（§0~§7，主条目 ~18k 字符，推荐新卡起步）**：
```
0 输出格式 → 1 完整 schema 树(速查+封闭性标注) → 2 路径反例 → 3 类型一致性
→ 4 数值范围 → 5 Enum 字段 → 6 联动规则 → 7 输出前自检
（详则拆独立条目：上限与清理 / 平板指令 / 组织系统…按 key 触发）
```

**中型版（10 章）**：输出格式 → UID 规范 → 自动计算字段(禁手写) → 关键联动 → 位置命名 → 物品键名纪律 → 路径速查 → 写入约定 → 字典池纪律 → 收尾校验。

**大型版（30+ 章多文件）**：按变量子树逐系统一章，配套 common/combat/relation/world/npc 多文件按第二 API 任务拆分。文件头 4 行职责声明是标配：

```
# [mvu_general] 通用变量更新规则
# 由 variable_update 任务读取。
# 包含：玩家系统 / 装备 / 技能 / …
# 不含：战斗/关系/世界规则（各自在专属文件中）
```

## 3. 字段五件套（规则的原子单位）

```yaml
生命值:
  path: /玩家/生命值
  type: number
  range: 0 ~ 生命上限
  default: 100
  check:                        # ← 规则的灵魂：触发时机清单
    - 受到伤害: 减少相应数值
    - 治疗恢复: 增加（不超过上限）
    - 自然恢复: 休息时每小时恢复5%上限
  关联同步: 必须同时更新生命状态
```

可选键：`enum`、`required: [...]`（建档必填）、`UID格式:`、`兼容字段:`（AI 常写的别名→正名映射）、`⚠️警告:`、`示例:`（仅在结构真的复杂时给一条值示例，不给成套模板）。复杂结构用 TS 类型字面量塞 `type: |-`（`Array<{时间: string; 维度: string; ...}>`）。string 类型可省 type；自解释变量不写规则；`_` 前缀只读不列。

## 4. UID 规范（三项目家规）

- 格式 `前缀_NNN` 纯数字序号（character_001 / item_005 / org_002 / unit_012…），**禁名字派生 key**（❌ /角色档案/张三）。
- InitVar 里按实体类型配 `/UID计数器/${前缀}`（小卡 9 类、大卡 40+ 类——按卡规模定）。
- 闭环：建档前按名字扫全库 → 命中用旧 UID replace 不 add → 不存在则 读计数器 → add 建档 → 计数器+1。写实发反例（"❌ 5个老孙头"）。
- 存量自愈：读到幽灵条目（残桩 key / 名称空 / 全默认）→ remove 重建。
- **字典 key 禁含英文句点**：JSONPatch 的 `/` 路径落库转内部点路径，`7.62×39弹` 会被 lodash 按 `.` 劈成 `"7"` 残桩。带口径/型号的 key 用无点形（`7,62×39弹`），真名写进对象的 `名称` 字段。

## 5. 建档门槛（宁缺勿滥，防实体爆炸）

负面清单（不建）：未报真名 / 功能性过场 / 本轮即离场 / 一次性物品 / 装饰名词。
正面清单（建，且一次写满）：已知真名 + 实质互动 / 会再登场 / 玩家明确深入意图 / 玩家要接取、决策、再交互的。
纯通用原则 LLM 会无视——**必须配 3~5 个具体锚点案例**（①NPC②势力③任务④区域⑤奖励各给一句判例）才有约束力。
新建实体禁占位默认（"待定"/空串）——要么不建，建就写满；未知字段填具体推断值。

## 6. 联动规则

- 关系变化必须双向（A→B 更新时 B→A 同步）；上下级双向（我.上级=X 时 X.下属+=我）；组织敌对/结盟双向。
- 数值状态同步协议：数值主、状态辅、阈值映射表硬绑定，`禁止操作: 不可单独修改状态而不修改数值`。
- 自动计算字段（schema transform 负责的）单独一章「禁止手写」，列清单。
- 多操作配方写成"联动更新速查"：`同伴加入: add /同伴档案/${新UID} + replace /当前队伍 + /UID计数器/comp +1`。
- 时间推进联动（改时间必改星期/倒计时/过期检查）。

## 7. 字典池纪律与清理（可独立成条目）

- 滚动窗口：动态类列表各配上限（动态≤20 / 情报≤8 / 归档池各≤15…），超限删最旧或合并。
- 当前快照类字段每回合 replace 覆盖，不累加。
- "叙事细节留正文，变量只存够下文承接的最短摘要。"
- 清理策略分级：必保留 > 优先保留 > 可合并 > 可删除；玩家明确保留 > 剧情连贯 > 数量上限。
- 大型卡拆独立 `[mvu_update]变量清理规则` 条目，或由自动清理脚本接管机械部分 + `setExtensionPrompt` 注入"超限提醒"让 LLM 做语义合并。

## 8. 反流水账（按语义类型识别高危字段）

变量只描述**当前状态/属性**，禁把正文事件原样塞进字段。判别口诀："**这条明天还成立吗？**"成立=状态可存；过时=事件，归历史摘要或正文。

高危字段**不看名字看语义类型**——检查你这张卡的 schema，凡是命中下面类型的字段，规则里都要单独标"禁流水账"并给 ❌/✅ 对照：

| 语义类型 | 存的应该是 | 流水账错误形态 | 正确形态 |
|---|---|---|---|
| 偏好/癖好/特长类列表 | 角色的抽象倾向 | 把上一场的具体动作/事件逐条 add（"38 条全是事件"） | 抽象名词，一条覆盖一类；同类合并 |
| 心境/情绪/状态快照 | 此刻怎么样 | 多个场景的感受用 + 号堆叠 | 每轮完全 replace，只写当下一刻 |
| 关系描述文本 | 当前是什么关系 | 两人交往史时间线 | 当前定性一句话 + 最新变化 |
| 个人总结/简介 | 这人现在是谁、在做什么 | 做过的事按时间罗列 | 身份+核心关系+当前阶段，定长压缩 |
| 外观/着装/形象描述 | 现在是什么样子 | 混入正在做的动作/所在位置/事件 | 纯静态描述，动作位置归各自字段 |
| 备忘/要事/日程类列表 | 对未来仍生效的约定 | 当日日记逐条 add 越积越长 | 只存未完成/未来生效项，完成即删或归档 |

根因提示：LLM 缺文风锚点时会把"这轮写得不错的细节"存进变量当记忆——治本配 plot 层文风指引（见 plot-style.md），治标靠本节类型表 + 每字段上限。

## 9. 输出前自检（收尾必备章）

多项目家规"A/B 双轴"扩展成六段：

```
A 路径合法（每段在 schema 树 / key 用 UID / 没漏中间层 / 没用废弃路径）
B 类型范围（数字不带引号 / 数组不写顿号串 / bool 非"是否" / pct 不越界）
C 联动同步（计数器 / 双向关系 / 数值状态 / 时间星期）
D 没整对象 replace（改单字段只 patch 该字段——replace 中间对象丢未列字段）
E 剧情-变量同步（正文出现的会反复出现的实体，变量里必须有；没有→本轮补建）
F 反流水账反思（对照 §8 类型表，本卡命中的高危字段逐个问"明天还成立吗"）
```

E 是最容易漏的：写完正文回看，"她去黑石科技上班"但 /组织 里没有 → 本轮 add + 计数器，绝不拖下一轮（下一轮必忘，剧情与变量永久脱节）。

## 10. 文档风格铁律

- 剥所有 markdown 装饰（`#`/`**`/`---`/`>`），yaml 缩进即结构；```yaml 围栏也不要，只有真 JSON 数据示例才配 ```json。
- 禁成套 JSONPatch 模板——LLM 会把模板里的 character_002/示例人名抄进真剧情。只写"哪些路径受影响"速查。
- 避开常见人名词汇做术语（"叶子路径"会被当角色名，用"单个字段"+路径示例）。
- schema 收紧后把 AI 实测错配写成「数据契约速查」❌/✅ 对照章（放开局协议里效果最好）。
- 快速参照表纠偏别名：`❌ 血脉名称 → ✓ 主血脉`。
