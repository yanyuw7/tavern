# 世界书组织与组卡部署

## 1. 前缀分层体系（前缀即路由，不只是标注）

| 前缀 | 消费者 | 说明 |
|---|---|---|
| `[InitVar]` / `[initvar]变量初始化勿开` | MVU 框架按条目名扫描 | **永远 enabled=false**；内容纯 YAML 初值 |
| `[mvu_update]` | 变量 AI（额外解析模式）或主 AI 变量段 | 变量写入规则；字段/路径/JSONPatch 只能在这层 |
| `[mvu_plot]` | 剧情 AI | 演绎/世界观/文风；出现字段名路径 = 立刻撤 |
| **无前缀** | **双发**（剧情 AI + 变量 AI） | "读取变量"类注入条目必须无前缀 |
| `[mvu_general/combat/relation/world/npc/common/genesis]` 等任务命名空间 | **仅前端**（disabled + order 10050+） | 状态栏按前缀读禁用条目拼给第二 API，永不进主提示词 |

### 1.1 前缀在两种更新模式下的行为（最容易误解的机制）

MVU 有两种变量更新方式（**用户在 MVU 扩展设置里选，不在卡里配**）：

| 模式 | 请求结构 | 前缀的作用 |
|---|---|---|
| 随 AI 输出 | 变量更新内联在主回复末尾，一次请求 | 前缀**只是人类可读的分类标注**，所有 enabled 条目都发同一个 AI |
| 额外模型解析 | 拆两次请求：剧情 AI 写正文 → 变量 AI 单独解析 | 前缀是**硬路由**——`[mvu_plot]` 只发剧情 AI、`[mvu_update]` 只发变量 AI、无前缀双发 |

**所以同一张卡在两种模式下行为可能完全不同**：随 AI 输出模式下一切正常，用户切到额外解析模式就炸——因为某条变量 AI 必需的条目挂了 [mvu_plot]。造卡时**默认按额外解析模式的路由纪律写前缀**，两种模式就都安全。

### 1.2 增殖 bug 实录（这套体系最贵的一课）

读取变量条目被错挂成 `[mvu_plot]读取变量` → 额外解析模式下**只发剧情 AI，变量 AI 完全看不到当前变量** → 变量 AI 扫不到"unit_001 = 某角色"已存在 → 凭空 `add` 同名幽灵档案（未命名/全默认值）→ 每轮重复，实体无限增殖。

- 症状识别：档案库里出现大量"未命名/默认值"的重复实体，且只在额外解析模式下发生。
- **修复手法：只改 comment 把前缀删掉（content / constant / 其他字段全都不动）**——前缀存在于条目名里，修路由不需要碰内容。
- 改任何条目前先自问一句：**"这条变量 AI 需不需要看见？"** 需要 → 绝不挂 [mvu_plot]。反向同理："这条剧情 AI 需不需要？"不需要（纯写入清单/字段规则）→ 挂 [mvu_update] 省剧情 AI 的 token。

### 1.3 前缀书写规范

- 前缀必须在 comment **最开头**，精确拼写 `[mvu_update]` / `[mvu_plot]`（**全小写**，实战体系统一写法）——别自创变体（`[MVU_update]`/`[mvu-plot]`/`[mvuupdate]` 都不会被按预期路由）。
- **同一功能拆两条**是标准操作：如"制造"功能 = `[mvu_plot]制造剧情指引`（怎么演）+ `[mvu_update]制造写入清单`（怎么 patch），两条挂**相同 key** 同触发、各自 order；绝不为省条目把两层塞进一条。
- 兜底写入规则（如"消息未标金额时按 X 扣费"）属于变量层，必须放 [mvu_update]——塞进 plot 的写入规则在额外解析模式下等于没写。
- `Mvu.isDuringExtraAnalysis()` 可在脚本里判断当前是否额外解析阶段（楼层 iframe 在该阶段抓不到 MVU 事件，见 scripts.md §3）。

## 2. 触发方式四种

| 方式 | 配置 | 用途 |
|---|---|---|
| 蓝灯常驻 | constant=true | 每轮必需：主规则/读取变量/文风/选择栏 |
| 绿灯 key | keys=[...] | 开局协议(`开始游戏`)/平板指令(`《关注》`)/题材系统(`组织,公司,帮派`)/制造(`制造,孵化`) |
| @@if 条件 | 内容首行 `@@if <EJS表达式>` | 按变量状态激活（如相关数值超阈值/相关计数>0 才发对应指南） |
| disabled 仅程序读 | enabled=false | InitVar / 前端专用规则库 / 弃用条目留档 |

pos/depth/order 惯例：绝大多数 `位置=角色定义前(pos 0), depth 4`；花活——守卫类 EJS `depth=0` 贴生成点、状态栏占位符条目 `pos=1`、选择栏 `order=9999` 压轴、前端专用段 `order=10050+`、官方标准件 `指定深度/系统/深度0/顺序14720`。同为 constant 的条目靠 order 排相对位置。

key 设计防误触发：**禁纯数字/≤2 字符拉丁 key**（"22"/"2B"/"Z1" 会命中属性栏数字——实测一次误注入 28 个不相干条目）；外部角色书用「姓名主键 + 世界名次键 AND_ANY」门控（keysecondary=[世界名] + selectiveLogic=0），且次键必须等于系统实际会写出的字符串（可静态验证：次键 ∈ 库的 ip 集合 ⟺ 门控生效）。

### 2.1 递归纪律（默认双禁，token 膨胀的隐形源头）

递归机制：条目被激活后，它的 **content 也参与关键词扫描**，可能连锁触发其他 keyed 条目（B 再触发 C…）。膨胀路径：常驻世界观条目提了招牌 NPC 名 → 每轮把该 NPC 的几千字人设条目拉进 prompt（玩家根本没提这个人）→ 人设条目里又提了组织名 → 组织条目也进来……

**默认全部双禁**（`prevent_incoming: true` 不可被其他条目激活 + `prevent_outgoing: true` 不激活其他条目），这也是官方模板标准件的配置。逐类看：

| 条目类 | 配置 | 理由 |
|---|---|---|
| 规则/指引/协议类（[mvu_update]/文风/演绎/选择栏） | **双禁** | 内容全是系统词字段名，outgoing 扫描纯误伤；常驻条目 incoming 无意义 |
| 读取变量 / 状态块协议 | **双禁** | 内容是变量 JSON——里面全是角色名地名，**不禁 outgoing 等于每轮全量递归**（最大雷） |
| 世界观常驻 | **双禁**（至少 prevent_outgoing） | 每轮都在，内容里的专有名词每轮扫别人 |
| NPC/地点 keyed 条目 | 默认双禁；**有意设计 lore 链才开** | 想要"提到地点→带出派系"的叙事深度时，仅给链上条目开对应方向，并用 `delay_until` 控制递归层级 |
| 外部角色书 | **双禁** | 外书误触发一半是递归背锅 |

原则一句话：**递归是"有意设计的 lore 链"专用特性，白名单式开放；任何"顺手不配"都是在给 token 膨胀留门**。生成世界书 JSON 时 `recursion: { prevent_incoming: true, prevent_outgoing: true, delay_until: null }` 是默认值，谁要开谁说明理由。

## 3. 标准件清单（一张 MVU 卡的世界书最小集）

1. **[InitVar]**（禁用）——空结构哲学：record 全 `{}`、数组全 `[]`，只有全局默认写实值（体力 100 等），由 LLM 剧情中逐步填充，防示例数据污染开局。UID计数器 全类目配齐。
2. **[mvu_update]变量更新规则**（常驻）——见 update-rules.md。
3. **读取变量 / 当前变量**（常驻，**无前缀**，`@@private`）——见 ejs.md。官方极简版是 `{{get_message_variable::stat_data}}` 宏直读全量；上规模必须换 EJS 过滤版。
4. **开局协议**（key=开始游戏）——叙事开局 [mvu_plot]（读 persona→定世界观→开场）与变量开局（建档 JSONPatch 清单/模板）分开写；也可把变量开局做成 disabled 条目由前端首轮注入。
5. **选择栏 / 输出末尾格式**（常驻 order 压轴）。
6. 可选：演绎指南（@@if）/ 文风指引（常驻）/ 守卫族（EJS）/ 题材系统条目（key 触发）/ 变量清理规则。

## 4. ST 世界书 JSON 硬规则（生成/修改时）

- **entries 的键必须 == 每条 entry 的 uid（字符串化）**：`{"entries": {"10": {"uid":10,...}}}`。用连续索引当键而 uid 是别的值 → 导入后条目列表**完全空白**（分页有数量但列表不显示）。正确构造：`{str(e["uid"]): e for e in entries}`。
- **字段集对齐同项目已验证可导入的文件**：不同 ST 版本字段有差异（`display_index` vs `displayIndex`、有无 `use_regex`/`name`）。以目标项目里确认能导入的 JSON 为模板全字段对齐，别跨项目套。
- 条目列表显示的"标题(备忘)"是 `comment` 字段不是 `name`（name 常为空）。
- 卡内嵌（`data.character_book.entries` 数组）与独立世界书（顶层 `entries` 对象）结构不同；两处同一条目内容同步用 byte-safe：`raw.split(JSON.stringify(old)).join(JSON.stringify(new))`（JSON 字符串转义与外层格式无关，两文件字节增量应相等）。
- 明确维护策略：独立世界书为主 or 卡内嵌为主，**别双写**（会分叉）。

## 5. 外部世界书处理（拿别人的书）

复验四件套跑脚本，不靠肉眼：① entry 总数 vs uid 唯一数对账 ② keyless 扫描（非常驻无主键=永不触发的哑火条）③ 跨条/跨书 dup 扫 ④ 关键字段抽样对照。
清洗流程：剔除非角色 lore（世界观/时间线/系统块）、逗号串主键拆独立 key、剪地雷 key、加门控次键、书名改易懂。多档变体（高清/轻量/加强三条同 key）默认只启用轻量。

## 6. 构建/同步管线（Python/Node 脚本骨架）

生成：`make_entry()` 全字段模板（对齐同源文件）→ `{str(uid): entry}` → 写后自检断言（键==uid / 字段集 diff / uid 唯一 / 逐条 token 报告）。
注入卡：按 SCRIPT_ID 或 comment 幂等去重后替换 → 写前 `.bak` → 写后 reload 复核（条目数 / 关键串 in content / 文件 size）。
双视图同步管线：按任务的源 md → 按条目的镜像 md → sync 脚本按 comment 匹配写回世界书 JSON → 再 sync 进卡内嵌。改源文件必须跑同步并提醒用户重导入。
锚点补丁工具：锚点命中数必须=1 才执行、JSON.parse 自检、.bak、双文件字节增量核对。临时脚本用完即删。

## 7. 交付与部署

### 7.1 部件交付（默认方式）

**默认不生成完整角色卡 JSON**——逐部件交付，每个部件独立文件 + 一句"放进哪里"，用户自己放进酒馆：

| 部件 | 交付形态 | 用户操作说明（附在部件末尾） |
|---|---|---|
| 世界书条目 | **首选可导入的独立世界书 JSON**（哪怕只有几条；必须过 §4 硬规则）；条目极少或用户要逐条审查时才给 txt+配置表 | "世界书页面导入此 JSON" |
| 脚本 | **每支一个可导入 .json**（完整字段集见 scripts.md §1.1，含 content/button/enabled） | "酒馆助手 → 脚本库 → 导入；改 schema 后重开对话" |
| 正则 | **每支一个可导入 .json**（11 字段完整模板见 regex.md §4） | "扩展 → 正则 → 按文件名编号顺序导入" |

**部件文件命名一律加两位序号前缀**（`01-xxx.json`、`02-xxx.json`…），序号 = 建议的导入/核对顺序：正则是硬需求（执行顺序=列表顺序，导错顺序=功能错乱）；脚本/条目文件编号则方便用户按序核对不漏项。顺序敏感的部件在名字里再加人话提醒（"必须在美化后"）。

**部件版本化（铁律 16 的落点）**：
- 每个文本类部件**内容首行/头注释**带版本（条目 content 首行 `变量更新规则 v1.2`、脚本头 `// xx脚本 v1.1`、schema 头注释版本块）——版本**不进**条目名/scriptName（那些是路由与去重的 key，必须稳定）。
- 任何部件修改 = 该部件版本 bump + 卡 `character_version` bump。
- **迭代交付必附变更清单**：`部件名 vX→vY：改了什么`——用户靠它知道这次要重新导入哪几个文件、跳过哪几个；没有版本对账的迭代交付等于让用户全量重导。
- 卡级可选：长线维护的卡配一份随包 CHANGELOG（发版时更新）。
| 状态栏 | dist 产物或托管 URL + 占位符正则 | "按 regex.md §2.C 配置三支正则" |
| [InitVar] | 条目内容 + ⚠️ 保持 enabled=false | 同世界书条目 |

好处：用户可逐个审查、不会覆盖已有卡内容、格式风险最小。

### 7.2 组装成卡（仅用户明确要求时）

用户明说"组装角色卡 / 打包成卡"才做。逐项过：

- 卡 JSON：character_book 条目齐 / [InitVar] disabled / 开场白含 `<StatusPlaceHolderImpl/>`（如有状态栏）且清除了 `<UpdateVariable>` 残块。
- tavern_helper.scripts：MVU bundle 加载器（国内/国外二选一启用）/ schema 注册脚本 / 功能脚本（检测/守卫等）。
- regex_scripts 标准件（一套 12 支多卡验证，**写法详见 regex.md**）：隐藏获取变量 / 去除变量更新（未闭合兼容）/ 思维链隐藏与折叠美化 / 变量同步流式双态美化 / 选择框美化+消除兜底 / 状态栏占位符→iframe 三支配合 / 第二 API 中间标签双重隐藏。
- 组装进**用户提供的现有卡**时：byte-safe 编辑 + 写前 .bak + 幂等去重（按条目名/SCRIPT_ID），绝不整卡重构。

### 7.3 部署验证（两种交付方式通用）

- 部署三套互不联动，交付时列全用户操作：重导世界书 / 重载 schema 脚本+重开对话 / 重 build 重绑 dist。
- 验证：开新聊天 → 查 stat_data 初值 → 输入开始游戏 → 查 AI patch 与落库 → 状态栏渲染。排查顺序永远 (a) AI 真实输出 (b) 变量真实存储 (c) 比对——不猜。
