# 纯文字卡专册

定义（本 skill 采用）：**无变量系统**的卡——没有 MVU/schema/脚本/前端，但**允许正则美化与文本式状态栏**。底线是"装上就能玩"：不依赖玩家侧任何扩展配置。

核心机制一句话：状态不存变量里，存在**聊天记录本身**——AI 每轮末尾输出结构化文本状态块，下一轮读到自己上轮写的块、更新后再输出，形成自我延续链。数据层是朴素文本（AI 写得稳），呈现层是正则渲染的 HTML（好看）。

## 1. 状态块协议（L1，核心）

世界书常驻条目（order 压轴）要求 AI 每轮回复末尾按精确格式输出，例：

```
<状态>
时间|第3天 傍晚
地点|城郊废弃教堂
同伴|莉娜:62(信任) 卡尔:38(警惕)
关键进展|发现了地下室的日记
持有|银质挂坠、半张地图
</状态>
```

设计规则：

- **字段少而稳**：6~10 个封顶。字段越多，AI 抄错/漂移概率越高；每加一个字段先问"没有它剧情会断吗"。
- **管道分隔单行字段**：`字段名|值`——最容易被正则捕获，也最不容易被 AI 写崩。避免自由段落式状态。
- **数值+档位双轨**：`62(信任)`——AI 忘了数字还有档位词兜底；档位给封闭枚举（陌生→警惕→中立→信任→亲密）让变化有轨道。
- **抗漂移关键句**：条目里写"**先照抄上一轮状态块，再只修改本轮变动的字段**"——把"凭记忆重写"变成"增量修改"，漂移率大幅下降。
- **变化速率约束**：好感单轮 ±10 以内、档位不跳级——mini 版变量更新规则，两三行就够。
- **开场白必须带初始状态块**（⚠️ 与 MVU 卡"开场白禁变量块"**相反**）——第一楼就有样可依，否则 AI 第一轮自由发挥出一个格式，后面全跟着歪。
- 上面是基础两区示例；完整结构还有**总结/摘要区**（长期记忆的命脉），见 §2.1——正式做卡用三区版。

## 2. 长期记忆四层

| 层 | 机制 | 说明 |
|---|---|---|
| L1 状态块链 | §1 协议 | 精确状态（数值/物品/关系）的唯一载体 |
| L2 深度裁剪 | 正则 promptOnly + **最小深度=2** | 旧楼状态块从 prompt 剥除，AI 只看最近 1~2 个——记忆连续、token 恒定、玩家回翻旧楼仍能看到当时面板 |
| L3 keyed 条目 | 绿灯 key=名字/事件名 | 静态设定的被动记忆 |
| **L4 摘要与总结** | **住在状态块里，随链传递** | 上下文滑出后的唯一历史，见 §2.1——纯文字卡的命脉机制 |

L2 的正则配置注意：基础 11 字段模板（regex.md §4）没有深度字段——需要时补 `"minDepth": 2, "maxDepth": null`（对应正则编辑器的"最小深度/最大深度"输入框）。

### 2.1 摘要与总结（L4 完整定义）

**先分清两个东西，职责不同不可混**：

| | 场景摘要 | 全局总结 |
|---|---|---|
| 性质 | **增量**事件账——"发生过什么" | **覆盖**状态照——"故事至今是什么局面" |
| 更新 | 场景/章节结束时**追加一条** | 每次更新**整段重写替换**，不追加 |
| 内容 | 事件结果 / 关系变化 / 得失 / 未兑现的承诺与伏笔 | 当前身份处境 / 关系网现状（每人一短语）/ 主线进度 / 未了之事 |
| 判别 | "这条**影响未来剧情**吗？"影响才记——过程细节/对话原文/氛围描写一律不记（反流水账的纯文字版） | "新玩家只看这段能接上故事吗？" |

**载体：并进状态块**（这是纯文字卡摘要能活下来的根本机制）——摘要/总结没有变量可存，写成独立楼层块会随上下文滑出而丢失；只有住在状态块里，才能靠"先照抄上一轮再修改"的链条每轮自动传递、在 L2 深度裁剪中天然存活。状态块扩成三区：

```
<状态>
时间|第7天 深夜
地点|港口仓库
同伴|莉娜:71(信任) 卡尔:30(警惕)
持有|银质挂坠、教堂钥匙
——
总结|逃亡中的前调查员，与莉娜结成同盟调查教团；卡尔身份存疑。当前：潜入港口查货运记录。未了：欠码头老板一个人情；教堂地下室尚未探完
摘要|①第1-2天:教堂发现日记，确认教团存在 ②第3-5天:莉娜坦白身世，结盟 ③第6天:卡尔说谎被识破，关系恶化
</状态>
```

**预算与滚动合并纪律**（防膨胀，写进协议条目）：
- 摘要区**限 3~5 条**：新条目挤入时，把最老的 2 条合并成 1 条"更早：……"（滚动归档）；合并时**必须保留**：未兑现的承诺、伏笔、关系定性变化；**可以丢**：日常互动、已闭环的事件。
- 总结区限 2~3 行；摘要+总结合计 ≤ 状态块一半篇幅。
- 整个状态块（三区合计）目标 ≤ 600 字符/轮——这是每轮的输出成本，超了就砍字段而不是砍纪律。

**触发时机**：用**叙事节点**不用楼数——"场景切换 / 时间跳跃 / 章节收束时更新摘要与总结"（AI 感知得到叙事节点，数不准楼数）。再配玩家指令兜底：起手页写明"输入【总结】可随时让 AI 整理摘要与总结"。

**双块方案（进阶，长跑卡可选）**：状态区每轮输出、记忆区（总结+摘要）拆成独立 `<记忆>` 块只在叙事节点重输出，两块正则分开裁剪（`<记忆>` 不裁或 minDepth 更深）——每轮省下摘要区的重复输出。代价：AI 忘记在节点重输出 `<记忆>` 块它就会滑出上下文，需在状态块里留一个"记忆块楼层|N"字段互相锚定。**默认单块，确认卡要跑超长线再上双块。**

**玩家主权兜底**（写进 README）："觉得 AI 忘事了，翻旧楼把关键事实用【记住：××】发一条"——AI 收到后写进下一轮摘要区。ST 原生 Summarize 扩展可作为额外保险，但不要依赖（质量不可控且需玩家自行开启）。

## 3. 状态栏呈现：按卡的气质设计，不套模板

数据呈现分离：AI 写朴素管道文本，markdownOnly 正则捕获字段替换成 HTML 面板。**不存在"最好看的通用样式"——状态栏的样子必须在做卡时根据这张卡的情况现想**。设计三问：

1. **这张卡的世界是什么质感？** 状态栏配色/字体/纹理从题材长出来：废土卡像 CRT 终端、古风卡像卷轴或手账、赛博卡霓虹等宽、现代恋爱卡像手机通知、克苏鲁像泛黄手稿——先答这个再写 CSS，而不是套一个万能渐变卡片。
2. **状态栏是"世界内物件"还是"系统面板"？** 拟物（游戏里真实存在的终端/手账/系统界面，可以玩沉浸细节）vs 中性面板（干净不抢戏，衬托正文）——由卡的叙事视角定。
3. **信息密度多少？** 数值玩法重的卡：表格+进度条+多字段；叙事为主的卡：一两行胶囊化摘要、`<details>` 默认收起——状态栏永远为正文服务，不跟正文抢注意力。

通用技术手段（怎么实现，不是长什么样）：

- 捕获的数字直接插 style 做进度条：`<div style="width:$1%">`
- `<details><summary>` 折叠，默认收起
- 颜色用 ST 主题变量（`var(--SmartThemeBodyColor)` 系）做基调、题材色做点缀——深浅主题都不炸
- 档位词映射颜色（每档一支正则或穷举 class）
- CSS 内联或类名带卡前缀，防污染整楼
- 反例：照搬别的卡的状态栏样式（气质错位）；堆动画特效（分散阅读）

## 4. 正则纪律（与 MVU 卡相反，最容易搬错）

| | MVU 卡 | 纯文字卡 |
|---|---|---|
| 状态/变量块对 AI | **隐藏**（promptOnly 剥除，变量走 MVU 管线） | **必须可见**（记忆链全靠它）——美化正则只能 markdownOnly，**绝不能 promptOnly** |
| 深度控制 | 一般不需要 | promptOnly 只用于 **minDepth≥2 的旧块裁剪** |

从 MVU 卡抄正则到纯文字卡时逐支核对三态——把"去除变量更新"那支原样搬过来，记忆链当场断掉且无报错。

## 5. 工作流（入口 C 展开）

1. 需求访谈（跳过数值/schema 类问题；多问一步：状态块要追踪哪几个字段、状态栏气质想要什么样）
2. 卡本体写作（card-writing.md；开场白**末页带初始状态块**）
3. 状态块协议条目（§1 格式 + 更新规则 + 抗漂移句，常驻 order 压轴）
4. 世界书纯 lore 组织（世界观/NPC keyed/文风指引——worldbook.md 触发方式与 JSON 硬规则、plot-style.md 全部适用；没有 [mvu_update] 层、没有读取变量条目）
5. 正则（§3 美化 + §2 深度裁剪 + 选择框美化消除等通用支——regex.md，注意 §4 纪律）
6. 部件交付（可导入 JSON + 序号前缀，照 SKILL.md 第 10 步）

## 6. 纯文字卡特有的坑

1. **状态块格式没给样例就上线**：开场白没带初始块 → AI 第一轮自造格式 → 正则捕不到 → 美化全哑。
2. **美化正则误开 promptOnly**：§4，记忆链断。
3. **字段贪多**：12+ 字段的状态块，十几楼后必然漂移（漏字段/编字段/值互串）。
4. **捕获正则写死字段顺序但 AI 换序输出**：要么条目里锁死顺序（"严格按此顺序"），要么每字段独立捕获（多支小正则比一支大正则稳）。
5. **好感等数值无速率约束**：一轮从 30 跳 90——状态块协议里必须带变化速率句。
6. **深度裁剪忘了配**：50 楼后 prompt 里躺着 50 个状态块，token 白烧且 AI 可能抄到旧块。
7. **摘要写成独立楼层块**：不在状态块链里 = 随上下文滑出即永久丢失。摘要/总结必须住在状态块里传递（§2.1）。
8. **摘要只加不合并**：没写滚动合并纪律 → 摘要区无限膨胀，每轮输出成本线性涨。限 3~5 条+最老两条合并。
9. **总结覆盖时丢伏笔**：全局总结每次重写，AI 会自然丢早期信息——合并/重写规则里必须点名"未兑现的承诺、伏笔、关系定性变化必须保留"。
10. **摘要记成流水账**：把过程细节/对话原文写进摘要——判别句"这条影响未来剧情吗"必须写进协议条目。
