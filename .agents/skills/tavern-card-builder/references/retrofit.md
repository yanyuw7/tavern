# 纯文字卡 → MVU 化改造

把没有变量系统的传统角色卡升级成 MVU+zod 卡。**本质是造卡工作流换入口**：第 1 步从"需求访谈"换成"读卡分析"，第 2~10 步走 SKILL.md 标准流程。本册只讲改造专属的部分。

## 0. 改造总原则

1. **原味保持是第一优先级**：原卡的人设、文风、剧情走向一个字不动。改造 = 在旁边**加**一层变量系统，不是重写这张卡。改了原文的改造是失败的改造。
2. **默认部件交付**（铁律 15 在改造场景加倍适用）：产出的是"新增条目 + 脚本 + 正则"部件包和一份"原卡需要动的最小改动清单"，用户自己合并。绝不直接吐一个改造后的完整卡覆盖原卡。
3. **最小可跑优先**：先四件套（schema + [InitVar] + 变量更新规则 + 读取变量）验证变量流转，跑通了再谈守卫/演绎指南/状态栏。一次性全家桶 = 排错时不知道哪层坏了。
4. 改用户提供的卡文件时：byte-safe + 写前 .bak + 幂等去重（见 worldbook.md §7.2）。

## 1. 第一步：读卡分析（替代需求访谈的第一轮）

通读卡的全部叙事字段：description / personality / scenario / first_mes / alternate_greetings / mes_example / 世界书全部条目。目标是回答三个问题：

**Q1 这张卡隐含哪些"值得变量化的状态"？**（按信号类型扫）

| 信号 | 例 | 变量化候选 |
|---|---|---|
| 数值暗示 | "好感度会影响她的态度""你身上有 500 金币" | 好感度 pct / 金钱 num |
| 状态描述 | "她现在很警惕""你受了伤" | 状态 enum / HP |
| 物品道具 | "她送你一把钥匙""背包里有..." | 物品栏字典 |
| 关系网 | 多角色 + 关系描述 | 人际关系字典（好感/阶段） |
| 时间地点 | "第一天""在酒馆" | 世界.当前时间/当前地点 |
| 进度机制 | 任务/阶段/解锁条件 | 任务字典 / 进度字段 |
| 隐藏信息 | "她有个秘密，熟了才说" | 秘密解锁 bool / 信任门槛 |
| 周期事件 | "每晚都会..." | 日程/计数器 |

**Q2 原卡的玩法核心是什么？**——变量系统必须服务它而不是喧宾夺主。恋爱卡的核心是关系 → 变量重心在人际关系+事件记忆；生存卡核心是资源 → 重心在物品+数值+时间。**不要给每张卡都套满编制**（角色档案+物品+任务+组织+战斗全家桶）——原卡没有的玩法不要发明。

**Q3 原世界书条目怎么分类？**——逐条打标：纯世界观/人设（保留，可加 [mvu_plot] 前缀或不动）/ 含玩法规则（可能要拆：叙事部分留 plot、数值规则进新的 [mvu_update]）/ 与变量冲突（如条目里写死"好感度用文字描述"——需要用户决策）。

产出：**变量树草图 + 原条目处理清单 + 玩法核心一句话**，进入访谈第三轮（复述确认）——改造不是免访谈，Q1 扫出来的候选**哪些要哪些不要、数值范围多少，仍然要问用户**（宁缺勿滥，变量越少改造越稳）。

## 2. 变量体系设计的改造特有考量

- **从小起步**：首版 schema 只收"玩法核心 + 剧情反复引用"的状态，10~20 个字段起步。改造卡最常见的死法是变量过多 → LLM 顾不过来 → 变量与剧情脱节 → 玩家觉得"还不如原卡"。
- **叙事字段不变量化**：外貌描写、性格散文这些原卡写得好好的东西，不要抽成变量（变量存状态，不存散文）。只有"会变的"才进 schema。
- **初值从原文提取**：first_mes 说"清晨的酒馆"→ InitVar 的时间/地点照填；description 说角色警惕 → 初始好感/信任给低值。[InitVar] 之外的空结构照 worldbook.md §3 空结构哲学。
- 原卡如果有"伪变量"（正文里用括号/状态栏文本手写的 HP 之类）→ 这是最强的变量化信号，直接映射，并加正则把旧的文本状态栏剥掉换成新状态栏。

## 3. 新增部件清单（标准改造包）

| 部件 | 说明 |
|---|---|
| schema 注册脚本 | 按 schema.md 写，封闭性决策照 §3 |
| MVU bundle 加载器 | 一行 import（国内/国外二选一） |
| [InitVar] 条目 | enabled=false，初值从原文提取 |
| [mvu_update]变量更新规则 | 精简版 §0~§7 起步（update-rules.md §2） |
| 读取变量条目 | **无前缀**；小卡用宏直读，实体多用 EJS 过滤 |
| [mvu_update]开局协议 | 若原卡开场白已固定剧情，开局协议改为"从开场白提取状态写入变量"而非重新生成开局 |
| 正则标准件 | 隐藏获取变量 / 去除变量更新 /（可选）变量同步美化——见 regex.md |
| （可选）状态栏 / 守卫 / 演绎指南 | 二期再加 |

## 4. 原卡的最小改动清单（唯一允许碰原文的地方）

- 开场白**末尾追加**（不改正文）：初始 `<UpdateVariable>` 块（若开局协议策略需要）或状态栏占位符。
- 世界书条目 comment 加前缀（content 不动）——按"变量 AI 需不需要看见"决策（worldbook.md §1.2）。
- 原条目里与变量系统直接矛盾的句子（罕见）→ 列出来**让用户决定**改不改，绝不擅自动。
- 除此之外原卡零改动。

## 5. 改造验证（比新卡多一步"回归"）

标准验证（worldbook.md §7.3）之外，加**原味回归检查**：改造后玩 3~5 轮，对照原卡感受——
- 角色说话还是原来的味道吗？（变量规则条目污染文风 = [mvu_update] 内容泄进了剧情层，查前缀）
- 正文有没有开始报数值/复述变量？（缺"不数值化叙事"约束，见 plot-style.md）
- AI 是否在乱建实体？（建档门槛没写锚点案例，见 update-rules.md §5）
- 变量真的在动吗？（观察器脚本上真证据，scripts.md §4B）

## 6. 纯文字卡模式（不 MVU 化，直接做传统卡）

用户明确不要变量系统时，读 **text-card.md 专册**——状态块协议（文本自延续链替代变量）、长期记忆四层、状态栏按卡气质设计、与 MVU 卡相反的正则纪律、裁剪版工作流全在那册。
